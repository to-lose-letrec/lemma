## Resolved

- **11. Welcome / introspection asymmetry.** Resolved by reframing rather than collapsing. `:welcome` carries *names only* — `:predicates {:core #{…} :extensions {pack-name #{…}}}`, same shape for `:verbs` — keeping `hello` cheap. The introspection verbs (`predicates`, `verbs`, `rules`, `capabilities`) return the rich detail (signatures, arities, types, cardinality, doc) on demand, world-scoped via the optional `:world` qualifier from §6. Added a `verbs` introspection verb (count §6: 22 → 23) and an `:event :verbs` response shape parallel to `:event :predicates`. Welcome and detail layers now have clearly separate roles.
- **Doc split.** `SPEC.md` is the wire-protocol contract; `SERVER.md` is the reference server-implementation guide (non-normative). §11 (Persistence) and §13 (Packs) moved out of SPEC.md to `SERVER.md`. SPEC's former §13 collapsed to "Pack-extended verbs and predicates" — six lines covering only the wire-observable bits (extensions advertised in `:welcome`, validated like core, `:pack/<symbol>` error namespace). Server-implementation deferrals (log compaction, pack version migration, adapter packs) moved to `SERVER.md` §3; protocol deferrals stay in SPEC §14. Section numbering renumbered: old §12 → new §11 (Import/export), old §13 → new §12 (Pack-extended verbs), old §14 → new §13 (Agent discipline), old §15 → new §14 (Open questions).
- **12. Proposal cancellation.** Added `cancel` verb (§6, §7): single-shape `(cancel #proposal "p-N")`, returns `{:event :cancelled :proposal #proposal "…"}`. Cancelling a proposal that doesn't exist / was already asserted / already cancelled / expired / belongs to another session returns `:error :reason :unknown-handle` — proposals are session-scoped, so the handle ceases to refer once the proposal is no longer pending. Purely advisory cleanup; the protocol does not depend on clients calling it. Verb count §6: 21 → 22.
- **1. Request-response correlation.** Solved via metadata, not new fields. §3 now permits `^{:request-id "…"}` on any client verb form; server echoes the value as `:in-reply-to` on the response event. Required for HTTP+SSE under concurrent requests; optional over Unix socket where FIFO ordering suffices. For stream-establishing verbs (`watch`, paginated `query`), `:in-reply-to` appears on the establishment event only — subsequent stream events correlate via their `#watch` / `#cursor` handle. Zero invasiveness: no new top-level envelope, no per-verb qualifier, no transport-specific machinery; just one optional metadata key the server already parses past.
- **3. Pack execution model.** v1 ships **logic packs only**. Packs are pure data: `pack.edn` declares predicates, rules, and verbs, where each verb body is an EDN form describing a parameterized query, rule application, or composition thereof. The server compiles verb definitions at load time and resolves `(pack-ns/verb …args)` invocations against the registered form, running them through the same engine as core `query` / `inconsistencies`. No host-language code path exists in v1 — the verb language has no syntax for filesystem, network, or cross-world operations, so the seal holds by construction without sandboxing.

  Trust model is pinned down in §12: pack installation is strictly out-of-band (no `install-pack` verb, by design); `meta.edn` names required packs but doesn't carry their bodies; opening a world whose declared packs aren't on disk is a hard `:error :reason :missing-pack` at welcome time. §11 mirrors this on the import side: `import` transports facts only, never packs/verb-bodies; a `:log` referencing an uninstalled pack fails with `:missing-pack`. §10 lists `:missing-pack` among reserved error reasons.

  Native-code "adapter" packs (parsing, hashing, format conversion) are deferred to v2 along with their sandboxing mechanism — §2 lists this as an explicit non-goal for v1, §14 carries the deferred design conversation.
- **2. Protocol-level error envelope.** `:event :error` added to §10 with reserved `:reason` set (`:malformed`, `:unknown-verb`, `:bad-args`, `:no-session`, `:unknown-handle`, `:missing-pack`, `:internal`) and a `:pack/<symbol>` namespace for pack-defined protocol errors. `:wrong-world` deliberately omitted — handles are world-scoped and a foreign-world handle is indistinguishable from a never-existed one.
- **6. Pre-hello surface.** §4 now states `hello` is the only verb permitted on an anonymous connection; anything else returns `:error :reason :no-session`. Client always initiates.
- **8. JSONL lossiness.** Struck entirely — `:jsonl` format and Cogito-interop line removed from §12; closing example workflow rewritten to drop the Cogito seeding scenario.
- **Lists are verb invocations, structurally.** `SPEC.md` §3 now states that any EDN list in a client message — top-level or nested — is treated as a verb form: top-level must resolve to an advertised verb, nested lists are a `:malformed` rejection. `SPEC.md` §10 updates `:malformed` to cover the nested-list case; the parser-side obligation is documented in `SERVER.md` §2 (two-phase decode-then-walk, walk-then-dispatch as the single enforcement boundary). Defense in depth — even though §2 already forbids any host-language execution path, the rule keeps a code-shaped payload from ever reaching a position the server might be tempted to interpret. Pairs with §11's import-transports-facts-only guarantee.
- **Extensional / intensional predicate distinction.** `SPEC.md` §10 adds `:intensional? bool` to the predicate-event metadata; a new bullet introduces the EDB/IDB terminology (Ullman 1988) and specifies that intensional predicates are produced only by rule derivation. The user-facing assertion paths (`assert`, `propose`, `import`) reject them with `:error :reason :bad-args :detail {:reason :intensional}`. `SPEC.md` §11 specifies that import/export round-trip extensional facts only — IDB is recomputed at import time from the imported EDB and the loaded packs' rules, keeping exports compact and stable across pack-version upgrades. `SERVER.md` §3.4 documents the three enforcement boundaries (assertion, import, storage).
- **25. Capabilities enum.** `SPEC.md` §10 reserves a small set of v1 protocol-flag capabilities (`:lemma/v1`, `:lemma/tls`, `:lemma/peer-cred`, `:lemma/cursor-pagination`, `:lemma/watch`, `:lemma/import`, `:lemma/export`) under the bare-keyword namespace; pack-defined capabilities namespace as `:pack-name/<flag>`. The set is open and additive. v1.x will introduce per-session capability gating (auth-tied) per §14.
- **26. Resource limits.** `SPEC.md` §10 specifies a `:limits` map in `:welcome` carrying server-policy caps: `:max-message-bytes`, `:max-fact-bytes`, `:max-facts-per-propose`, `:max-where-depth`, `:max-watch-buffer`. Servers MUST advertise each limit they enforce. New error reason `:limit-exceeded` carries the offending limit name, observed value, and max in `:detail`. Categories may grow additively in v1.x without breaking clients. Per-session quotas and per-pack rule-evaluation budgets deferred to v1.x per §14.
- **29. Local-socket peer authentication.** `SPEC.md` §3 specifies that servers MAY use SO_PEERCRED (or platform equivalent) to bind sessions to OS users; if used, the credential surfaces via `:lemma/peer-cred` capability and a `:peer {:uid N :gid N}` field in `:welcome`. v1 treats peer-cred as a hint, not a permission gate; full auth/authz is deferred to v1.x per §14, where peer-cred becomes a hook for the auth model.
- **30. TLS for HTTP+SSE.** `SPEC.md` §3 declares TLS strongly preferred for HTTP+SSE — non-localhost deployments SHOULD enforce TLS at the transport layer; clients SHOULD refuse to send credentials, proposals, or write verbs over non-TLS HTTP unless the operator explicitly opts in to plaintext. TLS 1.2 minimum, 1.3 preferred; certificate validation REQUIRED. The `:lemma/tls` capability flag advertises TLS-on-this-connection. Unix-socket has no analogous concern (filesystem permissions are the boundary).
- **13. Proposal expiry policy.** `SPEC.md` §7 specifies the default proposal TTL as **300 seconds** idle from creation, configurable per-world via `meta.edn` (the value used is reflected in every `:proposed` event's `:expires-at`). No proactive expiry event is pushed; clients learn at `assert` / `cancel` time via `:error :reason :unknown-handle` — expired and never-existed are indistinguishable to the client (per §4 session-scoping).
- **21. Cursor and watch TTL.** `SPEC.md` §8.1 specifies cursor idle TTL of **300 seconds**, refreshed on each `continue`; expired cursors return `:unknown-handle`. The cursor is a server-side bookmark (last-seen key + query metadata), not an open LMDB transaction (per `SERVER.md`), so cursor TTL is about server-memory housekeeping, not file bloat. `SPEC.md` §9 specifies watches as session-scoped — no idle-watch TTL, no idle-watch reaper; watches end with their session, with `unwatch`, or with the sustained-slow-consumer disconnect from the watch backpressure rules.
- **14. `tx-info`, `worlds`, `provenance` response shapes.** `SPEC.md` §10 now includes shapes for all three. `:tx-info` carries `:tx`, `:session`, `:timestamp`, `:ops` (canonical per-operation record matching the `log.edn` op format from `SERVER.md` §1.1, including implicit cardinality-driven retracts), `:metadata`, and optional `:proposal` (present iff tx came from `propose`/`assert` flow). `:worlds` lists each world with declared packs and head tx-id; pack bodies not included. `:provenance` carries `:tx`, `:session`, `:timestamp`, `:metadata`, and optional `:ref` (present iff request was for a fact via `#ref`; absent for tx-level provenance). `:metadata` carries the proposal-time `^{…}` annotations.
- **28. Single-axis bitemporality declaration.** `SPEC.md` §8 now states explicitly that Lemma's time model is transaction-time only — `:as-of` and `:between` reference `#tx`, never a separate valid-time axis. Domains that need valid-time semantics model them as ordinary predicates carrying date arguments. Datomic-trained readers should not assume an implicit valid-time qualifier.
- **31. Multi-pack rule interactions.** `SPEC.md` §12 now states that Datalog's monotonicity makes pack-load order irrelevant — when multiple packs contribute rules whose bodies reference shared predicates, all rules fire and the IDB is independent of load order. No rule-precedence mechanism, no shadowing, no per-pack isolation. Pack authors don't coordinate; they also own the cost of their rules (engine does not sandbox pathological packs at runtime). Operator recourse for a runaway pack is removal from the world's `meta.edn`.
- **4. Entity identity.** `SPEC.md` §5.3 specifies entity identity end-to-end. `#entity "<name>"` is a tagged literal whose value is a string treated as a name; structural identity (byte-equal name strings → same entity), per-world scope, no minting/creation/deletion verbs (entities exist by being referenced in facts). Equivalent entities remain distinct refs; substitution rules make them interchangeable but core does not canonicalize. Type tags `:id` and `:set` in pack predicate declarations enforce rule-body shape at pack-load time, but the runtime does not maintain an entity-kind registry — they are semantic, not runtime-enforced; domain packs that need runtime kind enforcement implement it via predicates and rules. Empty names rejected (`:bad-args :detail {:reason :empty-entity-name}`); all other EDN-string-valid contents accepted (whitespace, namespaces, Unicode); clients namespace their own names when collision risk matters. Self-equivalence (`equivalent #entity "a" #entity "a"`) accepted as vacuous no-op. Cross-world via export/import: every entity reference treated as new in the receiving world (isomorphic-but-distinct), consistent with import's tx-id reassignment.
- **10. Log framing.** `SERVER.md` §1.1 specifies the format as a **sequence of top-level EDN values** with arbitrary whitespace between records (pretty-printed, blank-line-separated by convention). No special delimiter or length-prefix — `clojure.edn/read` consumes the file as the EDN stream it is, preserving the `cp log.edn` admin invariant. Per-record shape: `{:tx-id :session :timestamp :ops [...] :metadata}` with `:ops` carrying both explicit operations and implicit cardinality-driven retracts. fsync per tx for durability. Crash recovery via **forward parse**: read records one at a time tracking byte position; on parse failure, truncate at the position of the last successful read. Cost is O(log size) but the server reads the whole log on boot anyway, so the framing adds no startup cost. Future optimization: checkpoint last-good-position in `meta.edn` for log-tail-only recovery (additive, not v1). Tx-ids are monotonic but not gap-free across crashes. Index rebuild on detected log/index mismatch falls back to full replay from log.
- **9. Watch backpressure.** `SPEC.md` §9 specifies the hybrid 1+3 policy: per-watch bounded buffer (server default 1000 events), `:watch-gap` reconciliation events when the buffer fills (carrying `:missed-since` / `:missed-until` / `:event-count`), with the watch resuming normal delivery as the consumer drains. The consumer reconciles via `query :where <pattern> :between [...]`. Sustained slow consumers (server-policy threshold across a sliding window) get disconnected with `:watch-closed :reason :slow-consumer`; the `#watch` handle becomes `:unknown-handle` after close, and the client re-establishes via fresh `watch` / `watch-pattern`. Establishment is deltas-only — no initial snapshot; clients needing ground truth issue a synchronizing `query` at subscription time. §10 adds `:watch-gap` and `:watch-closed` event shapes; `:watch-event :type` enumerated as `:added | :retracted`.
- **22. `continue` on expired/closed cursor.** Resolved as part of cursor-TTL work (item 21): `SPEC.md` §8.1 specifies `continue` against an expired cursor returns `:error :reason :unknown-handle`. Same handle-disappearance semantics as expired proposals; the cursor's TTL window having elapsed is indistinguishable from never-existed at the wire layer.
- **5. Cursor stability.** `SPEC.md` §8.1 specifies a default ordering for non-aggregated queries: **(tx-id ascending, ref-id ascending)** of the underlying facts that satisfied `:where`. This is the order facts entered the EDB, stable across runs against the same `:as-of` snapshot, making `#cursor` pagination reliable without the client specifying an order. Aggregated queries cannot be paginated — they return all groups in a single `:result` and `continue` on an aggregated cursor errors as `:bad-args :detail {:reason :aggregated-cursor}`. v1 omits `:order-by`; clients post-process if they want a different sort. Adding `:order-by` is additive, deferred to v1.1.
- **7. Affordance executability.** `SPEC.md` §10 now states the `:verb` field of an affordance is a complete, ready-to-send s-expression — paste-able directly, no template substitution. If the server cannot construct a complete invocation (missing args the agent must decide), it omits the affordance rather than emitting a template. Templates with placeholder syntax are deliberately out of scope; the server rarely has enough context to template usefully and placeholders introduce ambiguity. `:hint` is a short human-readable string explaining the suggestion.
- **23. `retract :where [...]` matching no facts.** `SPEC.md` §7 now states this is a successful no-op returning `{:event :retracted :refs []}` *without* a `:tx` field — the empty transaction is elided rather than burning a tx-id on a write that didn't change the world. §10 generalizes this: the presence of `:tx` in any commit-event signals "a tx was written"; absence signals "the world did not change." The same elision applies to no-op `propose`/`assert` (cardinality-replace where `v_old = v_new`).
- **15. Query language grammar.** `SPEC.md` §8.1 now defines the query grammar end-to-end. Variables are `?`-prefixed symbols; constants are tagged literals / strings / numbers / booleans / `nil` (bare unprefixed symbols in argument position are `:bad-args`). Fact patterns are vectors with a *symbol* head (`[member-of ?x ?s]`); operator clauses are vectors with a *keyword* head (`[:> ?age 30]`, `[:not [...]]`, `[:and [...] [...]]`, `[:or [...] [...]]`). The grammar dispatches on head type. Reserved comparison operators in v1: `:=`, `:!=`, `:>`, `:<`, `:>=`, `:<=`. Reserved logical operators: `:not` (NAF, per §8), `:and`, `:or`. Pack-defined operators namespace as `:my-pack/op`; bare keywords reserved for core. `:find` is a vector of variables and aggregations (`[:count ?x]`, `[:sum ?v]`, `[:max ?v]`, `[:min ?v]`, `[:distinct ?x]`); always returns a relation (no scalar/tuple variants). Aggregations group by non-aggregated `:find` vars. Predicate-variable patterns (`[?p ?x ?y]`) allowed but require at least one variable bound elsewhere. Same grammar reused by `watch-pattern :pattern [...]` and `retract :where [...]`. Visibly distinct from Datomic / Datalevin (predicate-first, keyword-headed operators, no nested lists); a Datomic reader notices the differences immediately and does not assume Datomic semantics carry over.
- **24. `watch-pattern` grammar.** Resolved as a side effect of item 15: `watch-pattern :pattern [<clause> …]` accepts the same `:where`-style clause vector defined in §8.1. The watch fires on assert/retract events that change the matching set.
- **18. Cardinality and uniqueness enforcement.** `SPEC.md` §7 now defines a five-step write-time check order (type → intensional → unique → cardinality-self-conflict → cohesion), with `:cardinality :one` implementing replace-on-conflict (implicit retract atomic with the new assert, post-state seen by the cohesion check) and `:unique? true` implementing reject-on-conflict (no replace; engine cannot infer which subject to remove). The two flags are orthogonal — `:cardinality` is per-subject, `:unique?` is global to the predicate's value. `:asserted` events carry `:retracted-refs` for cardinality-driven auto-retracts; no separate `:retracted` event is emitted for them. `:rejected :reason` enumeration extended in §10 with `:cohesion`, `:unique-conflict`, `:cardinality-self-conflict`, `:foreign-proposal`, `:stale-proposal`, `:orphan-referent`. No-op writes (cardinality-one assert with `v_old = v_new`) elide both ops; `:asserted :refs []` returns without burning a tx.
- **20. World creation path.** `SPEC.md` §4 and `SERVER.md` §3.6 now state that world creation is strictly out-of-band, parallel to pack installation. The operator places `meta.edn` (and an empty `log.edn`) under `~/.lemma/worlds/<name>/`; no client verb creates, deletes, renames, or otherwise modifies the world inventory. The `worlds` introspection verb lists; that is the full wire surface for world management. Same trust rationale as pack installation: world creation establishes shared state with cohesion semantics and pack dependencies, must be an operator decision.
- **19. Retract bypasses cohesion.** `SPEC.md` §7 now states explicitly that `retract` is not cohesion-checked: a retraction MAY leave the world inconsistent, and the violations it introduces surface via `inconsistencies` for the next `propose` to see. The asymmetry vs. `propose`/`assert` is deliberate — retraction is a corrective action that sometimes needs to break a temporarily-cohesive state to reach a better one; gating it behind cohesion would create dead-ends a world cannot recover from.
- **17. `#violation{…}` shape.** `SPEC.md` §5.2 now defines the violation shape: `{:rule <pack-qualified symbol> :anchor <entity> :bindings {?var value, …} :message <string>}`. Violations appear in `:rejected` events from `propose`/`assert` and in `:inconsistencies` events; the `:violations` field is always a vector (one entry per rule firing; multiple firings or multiple bindings produce multiple entries). Pack manifests that derive `inconsistent` may carry `:name` (becomes the rule symbol's suffix) and `:message` (template with `?var` substitution; entity values render as bare names, primitives as EDN literals). Both optional with fabricated fallbacks. Premises (the body facts that triggered the rule) are intentionally excluded from v1; agents recover them via `query` from the bindings if needed. This is additive — `:premises` can be added in a later revision without breaking clients.
- **16. `#fact{…}` shape.** `SPEC.md` §5.1 now defines the inline fact shape: `#fact{:predicate <symbol> :subject <s> :object <o>}` for binary predicates, `:object` omitted for unary, `:args [...]` for 3+-arity (uncommon, pack-defined). Predicate names are symbols, namespaced for pack extensions. Arity mismatch is `:bad-args`; an unknown predicate is `:bad-args :detail {:reason :unknown-predicate}`; an intensional predicate is rejected per §10's intensional rule. EDN metadata via `^{…}` is preserved as proposal-time provenance and surfaces via `provenance`.
- **27. EDN dialect.** `SPEC.md` §3 now pins the dialect to the edn-format public spec (https://github.com/edn-format/edn) with `clojure.edn` as the reference implementation for ambiguities. `clojure.edn/read` (not `clojure.core/read-string`) is mandated — the former is safe by default and rejects code-shaped reader extensions (`#=`, var literals, function literals, regex literals); the latter executes them. A second bullet specifies tagged-literal closure: the accepted tag set is exactly §5's table plus any advertised by loaded pack extensions, with unknown tags rejected as `:malformed` (stricter than `clojure.edn`'s default `tagged-literal` preservation). Implementations install explicit readers per known tag plus a `:default` reader that throws.
- **Negation flavor.** `SPEC.md` §8 specifies failure-as-negation (NAF) over stratified rules. A `not` clause inside `:where` succeeds when the negated pattern cannot be derived from the EDB and the loaded packs' rules; pack rule sets are stratified at load time so NAF has well-defined fixed-point semantics. There is no classical negation on the wire — the protocol does not assert `¬P` as a fact.
- **Engine vs. wire layering.** `SPEC.md` §8 makes explicit that the query language is Lemma's, not the underlying engine's. Clients only see Lemma's `:find` / `:where` grammar; the server compiles to whatever the configured engine accepts (the reference server uses Datalevin; see `SERVER.md`). No verb passes engine-native syntax through. This keeps the wire surface narrow and engine-agnostic, and a future server backed by a different rule engine remains protocol-conformant. Combined with §2's "no host-language execution path" non-goal and §3's lists-only-at-top rule, the wire surface is fully constrained to Lemma's own vocabulary.
- **Substitution rules and the Leibniz pattern.** `SERVER.md` §3.5 outlines how substitution rules are authored and executed: per-predicate-and-position enumeration of `∀P. (a ≡ b) → (P(a) ↔ P(b))` because Datalog cannot quantify over predicates. Type alignment is required (substitution into a `:set`-typed position needs a same-typed equivalence relation; `:id`-typed `equivalent` does not enumerate into set positions in core). Closure size is *O(n)* per equivalence class per containing relation; pathological cases bound at design time. Substitution-derived facts are intensional; the `:intensional?` invariant flows transitively through the rule body. Core uses three substitution rules covering `implies` (both arguments) and `member-of` (first argument); set-typed substitution is left to domain packs that need it.
